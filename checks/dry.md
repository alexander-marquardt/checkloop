---
id: dry
label: "DRY / Eliminate Repetition"
---

Find repeated or near-repeated logic. Extract shared helpers, base classes, or utility modules to eliminate duplication. Consolidate config values or magic numbers into constants. Where a module mixes multiple concerns (e.g. data models, API serialization, and validation in one file), consider extracting each concern into a focused module — but only when the separation makes each piece independently testable or reusable. Ensure each concept has a single canonical home in the code. Do NOT extract helpers for code that is only duplicated 2-3 lines or used in only 2 places — three similar lines is better than a premature abstraction. Do NOT change observable behaviour — only reduce repetition.

Do NOT propose extraction for near-identical *config files* where the variance between them is the intended axis of comparison — for example, dev/staging/prod settings files, Kubernetes overlays under `overlays/{env}/`, Terraform workspace tfvars, multiple `execution_plans/*.toml` or `pipelines/*.yaml` files whose whole purpose is to differ on a small set of dimensions, fixture files that intentionally pair "valid" and "invalid" variants of the same shape, or per-platform manifest files. The repetition in those cases is the *legibility* — a reader compares two files side-by-side to understand what changes between modes. Collapsing them into a base + diff (a Jinja template, a YAML anchor chain, a Kustomize patch tree) shifts the cost from "scanning two short files" to "mentally rendering a templated one", and that is usually a worse trade. Apply this carve-out when (a) the files live in a directory whose name or layout signals an axis of variance (`environments/`, `overlays/`, `plans/`, `presets/`, `profiles/`, `configs/{name}/`), (b) the variance is small relative to the file size, and (c) the files are read by humans as a set. Flag the duplication in your report so the maintainer can decide, but do not extract.

The same principle applies to **parallel implementations that share a surface across a transport or process boundary** — typically an in-process *adapter* paired with an *HTTP proxy* (`engine_adapter` + `engine_proxy`), a library API paired with its REST/gRPC wrapper, a sync client paired with an async client, or a local SDK paired with a remote SDK. These pairs intentionally repeat method signatures, argument shapes, and return types so they remain swappable — the duplication is the *contract* that lets a caller switch transports without changing call sites. Extracting a "shared base class" or a code-generated stub usually trades a small, obvious duplication for a heavier abstraction (a runtime registry, a generated client, a multiple-inheritance hierarchy) that adds more friction than the duplication ever cost. Recognise the pattern by these signals: (a) the two modules expose the same method names with matching signatures, (b) one delegates to in-process code while the other delegates to a network/IPC client, (c) they are imported by callers selectively based on deployment topology, and (d) the project's CLAUDE.md / AGENTS.md / CONTRIBUTING.md documents the parity as a deliberate design. When the pattern is in place, do not propose extraction or a shared base; report the duplication as intentional and move on. The same applies to any "library + HTTP" parity (e.g. a Python module callable both directly and via a REST endpoint that mirrors its interface) — the parity itself is the load-bearing property.

The same carve-out covers **test files that pair modes or variants of one module against the same fixtures**. `test_search_strict.py` + `test_search_relaxed.py` testing the same module's two modes against shared fixtures, `test_<feature>_valid.py` + `test_<feature>_invalid.py` covering both halves of an input contract, `<feature>.unit.test.ts` + `<feature>.integration.test.ts` covering the same surface at different levels — these are *legibility*, not duplication. A reader opens both files side-by-side to understand what changes between modes; merging them into a parametrized super-suite obscures the variance that is the whole reason for the split. Apply this carve-out when (a) the filename pair signals the variance axis (`_strict`/`_relaxed`, `_valid`/`_invalid`, `_happy`/`_error`, `_v1`/`_v2`), (b) the bodies are similar but the *expected outcomes* differ on the named axis, and (c) the test files exercise the same source module from different angles. Flag the surface duplication if you must, but do not merge.
