id: api-design
label: "API Design & Consistency"
---

Review public APIs (REST endpoints, library interfaces, CLI commands, exported functions) for consistency and usability. Check for: consistent naming conventions, predictable parameter ordering, appropriate HTTP methods and status codes, consistent error response formats, proper use of pagination, versioning where needed, and idempotency of mutating operations. Do NOT rename or remove the **external contract** — endpoint paths, HTTP methods, request/response field names, response shapes — without a versioning or compatibility path; those break live consumers. This does NOT freeze the *internal* representation: when an internal identifier is overloaded or misleading (see the readability check), disambiguate it in code and keep the existing wire name as an explicit alias — e.g. renaming an internal `direction` field to `polarity` while the response still serializes `direction` is the correct move, not a breaking change. Focus on parameter validation and error response consistency. Fix inconsistencies and document any breaking changes to the external surface.
