Improve naming (variables, functions, classes, fields, enum values, config keys). Distinguish two kinds of naming work. **Cosmetic renames** — swapping one clear name for another equally-clear name out of preference — are churn: large diffs through hot paths for no correctness gain. Do NOT make them. But two naming defects are *correctness* hazards, and the fix for each IS a rename — make it, and apply it to every call site and every layer in the same change (a half-applied rename is worse than none):

- **A name that lies.** The identifier asserts something its value contradicts: a `direction` field that holds `"include"` for a negation, an `is_empty` that is true when full, a top-level `field` attribute that actually holds an operator-suffixed lookup key. A misleading name is read as truth by every caller until one of them trusts it straight into a bug. Rename it to match what the code actually computes. If the name lies because the *value* is the bug (not the name), do NOT paper over it by renaming around the lie — flag it loudly for correction.
- **A name that is overloaded or inconsistent.** Either one concept carried under several disagreeing names across records or layers (`direction` here, `polarity` there, a `__not` suffix elsewhere), OR a single name reused for genuinely different things by context or nesting (a top-level `field` meaning the lookup key while a nested `x.field` means the clean value). The rule both ways: one concept → one name, everywhere; one name → one meaning, everywhere. When a concept already has a name, reuse it; if the existing name is wrong, rename it rather than adding a synonym beside it.

Renaming an internal identifier while preserving the external wire / public name as an explicit alias is a clarity fix, not a behaviour change (see the api-design check). Break up any function that does more than one logical thing, or that requires scrolling to read in full. Prefer small, named functions where the name removes the need for a comment.

A file's length alone is not a reason to split it. Split a file only when (a) it carries genuinely separate concerns that are read independently — e.g. data models in the same file as their HTTP serialization and their validation rules — AND (b) callers of the file will benefit from the smaller import surface, AND (c) the split does not require touching more than a handful of call sites. Files in the 600–900 line range with a single coherent concern stay as one module. Splitting test files purely to reduce line count is forbidden — a 700-line `test_<module>.py` that is one cohesive test of one module is correct as-is; splitting it creates orphan-file blame churn without making tests easier to find. Before proposing a split, name the specific reader who is currently struggling and the question they are trying to answer; if you cannot, do not split.

Do NOT add docstrings, comments, or type annotations to code you didn't change. Do NOT add module-level, class, or function docstrings unless the code is genuinely confusing without one. If the name and signature tell the story, a docstring adds nothing — leave it out. Do NOT add inline comments that restate what the code does. Only comment on non-obvious design decisions or surprising behaviour. Ensure consistent formatting. Do NOT change any behaviour — only improve clarity.
