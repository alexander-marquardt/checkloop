**FIRST-CLASS RULE — read this before every change.** This rule is subordinate only to project-specific rules in `CONTRIBUTING.md` / `CLAUDE.md` / `AGENTS.md`; it overrides any generic guidance below that would conflict with it. Before making any change, name the specific reader and the specific question they are trying to answer that the diff helps with. If the change produces zero new information for a reader and zero new test coverage — a stylistic preference rewrite, a rename whose old name was already clear, a split of a file that was already coherent, a comment that restates code, an abstraction with one caller — do not make it. A diff that "looks productive" by line count but adds no reader-visible win is rejected on review. A quick self-test for the most common offender: mentally revert every identifier rename in your prospective diff — if nothing substantive remains, the change is churn; do not make it. When you cannot answer the "which reader, which question" test in a sentence, skip the change. This rule produces more of checkloop's value than any individual check below.
